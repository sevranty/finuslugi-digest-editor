# finuslugi-digest-editor

`finuslugi-digest-editor` - личный навык для доказательной проверки продуктовых дайджестов Финуслуг

Навык работает только по предоставленным материалам и активной редполитике. Он сохраняет точные числа, знаки, единицы, периоды и статусы, отмечает непроверяемые фрагменты и не заполняет пробелы догадками.

## Для кого

- редакторы
- продуктовые команды
- дизайнеры
- аналитики
- владельцы дайджеста

## Три режима

| Режим | Когда включается | Что проверяет | Результат |
|---|---|---|---|
| Вычитка | «вычитать», «провести ревью», «проверить по редполитике», «валидировать» | результат, пользовательскую пользу, метрики, структуру, теги и Tone of Voice | таблица замечаний и приоритеты |
| Теги | «подобрать теги», «сформировать теги», «проверить теги» | количество, доказуемость, конкретность и соответствие карточке | таблица тегов или аудита |
| Сравнение | «сравнить», «сопоставить», «найти расхождения» | структуру, факты, числа, статусы, причинность, смысл и теги | вердикт и таблица расхождений |

Если явно запрошено несколько режимов, порядок фиксирован:

```text
сравнение -> вычитка -> теги
```

Несколько приложенных файлов сами по себе не включают сравнение.

## Поддерживаемые входы

- текст
- DOCX
- PDF, включая сканы
- TXT и Markdown
- презентации
- скриншоты и изображения
- таблицы

Для каждого файла навык строит доказательную карту: формат, роль, расположение разделов и карточек, утверждения, метрики, статусы, теги и нечитаемые места.

## Форматы результата

### Вычитка

| # | Раздел | Что не так | Тип проблемы | Серьезность | Почему это проблема | Как поправить |
|---:|---|---|---|---|---|---|

После таблицы навык называет самые проблемные разделы, приоритет исправлений, системные ошибки и сильные разделы. Если нарушений нет, сообщает об этом прямо.

### Формирование тегов

| # | Раздел | Краткое содержание карточки | Теги или диагностические маркеры | Логика выбора |
|---:|---|---|---|---|

### Аудит тегов

| # | Раздел | Исходные теги | Вердикт | Что нарушено | Рекомендуемый набор | Логика замены |
|---:|---|---|---|---|---|---|

По базовому контракту формируются три доказуемых тега. При нехватке смыслов используется маркер `— (недостаточно данных)`.

### Сравнение

Сначала навык явно назначает роли файлов, затем возвращает:

- вердикт `Соответствует`, `Частично соответствует` или `Не соответствует`
- количество проверенных смысловых единиц
- количество критичных, средних и низких расхождений
- таблицу только по найденным расхождениям
- корректно перенесенные элементы
- первоочередные исправления
- непроверяемые фрагменты

## Обязательный workflow

1. Определить только явно запрошенные режимы
2. Прочитать `references/file-handling.md`
3. Построить доказательную карту материалов
4. Загрузить протоколы выбранных режимов
5. Проверить текст, числа, знаки, единицы, статусы и расположение карточек
6. Выполнить режимы в порядке `сравнение -> вычитка -> теги`
7. Сверить вывод с исходными материалами и форматом протокола
8. Выдать полный пользовательский результат

## Роли файлов

При сравнении по умолчанию:

- файл 1 - источник
- файл 2 - проверяемая версия

Пользователь может назначить роли явно. Если порядок нельзя определить надежно и от него зависит вывод, навык задает один уточняющий вопрос.

При вычитке нескольких представлений одного дайджеста навык выбирает основную проверяемую версию, а остальные использует для контроля целостности. Материальный конфликт фиксируется как риск без выбора «правильного» варианта без первоисточника.

## Активная редполитика

По умолчанию используется `references/redpolicy.md`.

Пользовательская редполитика:

- заменяет встроенные содержательные критерии
- не меняет процедуру, доказательность, категории серьезности и формат протокола
- не смешивается с внешними редакторскими критериями
- при конфликте количества тегов без заданного приоритета вызывает один уточняющий вопрос

## Ограничения

- без независимого источника проверяется внутренняя согласованность, а не истинность во внешнем мире
- исходные файлы не изменяются без отдельной просьбы
- допустимая стилистическая переформулировка не считается расхождением
- ошибки извлечения не выдаются за ошибки исходного файла
- нечитаемые фрагменты не восстанавливаются по смыслу
- общие заголовки и декоративные подписи не считаются карточками
- несколько файлов не запускают сравнение автоматически

## Personal skill package

В пакет входят ровно семь runtime-файлов:

```text
finuslugi-digest-editor/
|-- SKILL.md
|-- agents/
|   `-- openai.yaml
`-- references/
    |-- comparison-protocol.md
    |-- file-handling.md
    |-- redpolicy.md
    |-- tagging-protocol.md
    `-- validation-protocol.md
```

Repository-only файлы не входят в personal skill package.

### Сборка

```bash
python3 scripts/build_package.py --output dist/finuslugi-digest-editor.zip
python3 scripts/validate_package.py dist/finuslugi-digest-editor.zip
```

Builder создает архив, JSON manifest и SHA-256 sidecar. Архив имеет один корень `finuslugi-digest-editor/` и содержит ровно семь runtime-файлов.

### Smoke-проверка установки

```text
Используй @finuslugi-digest-editor. Проверь этот текст по встроенной редполитике: «Запустили улучшения и получили отличный результат».
```

Ожидается режим вычитки, отсутствие выдуманной метрики и конкретная правка слабого результата.

## Проверенные примеры

### Вычитка

```text
Используй @finuslugi-digest-editor. Вычитай приложенный PDF по встроенной редполитике. Проверь карточки, метрики, структуру, теги и Tone of Voice. Сортируй замечания по критичности и не переписывай документ целиком.
```

### Формирование тегов

```text
Используй @finuslugi-digest-editor. Сформируй по три доказуемых тега для каждой самостоятельной карточки приложенного DOCX. Не назначай теги общим заголовкам и не заполняй недостающие смыслы догадками.
```

### Аудит тегов

```text
Используй @finuslugi-digest-editor. Проверь существующие теги: количество, доказуемость, конкретность и соответствие тексту карточек. Для несоответствующих наборов предложи замену.
```

### Сравнение

```text
Используй @finuslugi-digest-editor. Сравни файл 1 как источник и файл 2 как проверяемую версию. Найди потери, добавления, конфликты чисел, изменение статусов, причинности и смысла. Покажи только расхождения и точные значения.
```

### Все три режима

```text
Используй @finuslugi-digest-editor. Сначала сравни исходный DOCX с финальным PDF, затем вычитай PDF по приложенной редполитике и после этого проверь существующие теги. Раздели результаты по режимам.
```

## Назначение файлов

- `SKILL.md` - точка входа и маршрутизация режимов
- `agents/openai.yaml` - UI metadata и prompt по умолчанию
- `references/**` - file handling, redpolicy и протоколы
- `evals/**` - синтетические fixtures и expected results
- `release/package.json` - единственный package allowlist
- `scripts/**` - валидация и сборка
- `tests/**` - positive и negative self-tests
- `docs/**` и `AGENTS.md` - сопровождение и orchestration
- `assets/**` и `design/**` - repository-only визуальные материалы
- `README.md`, `CHANGELOG.md` и `LICENSE` - repository-only упаковка

## Локальная валидация

```bash
git diff --check
python3 scripts/validate_fixtures.py
python3 scripts/validate_repository.py
python3 -m unittest discover -s tests -p 'test_*.py'
python3 scripts/build_package.py --output dist/finuslugi-digest-editor.zip
python3 scripts/validate_package.py dist/finuslugi-digest-editor.zip
```

Проверяются:

- runtime 7/7 и package allowlist
- frontmatter и ссылки `SKILL.md`
- metadata `agents/openai.yaml`
- 7 regression cases и 11 SHA-256 записей
- README links и MIT License
- asset provenance, hashes и размеры
- deterministic ZIP, manifest, checksum и install smoke
- private keys, абсолютные локальные пути и временные файлы

## Изменения

Изменения поведения навыка вносятся отдельно от упаковки.

1. Создать Issue с целью, scope, non-goals и acceptance criteria
2. Создать отдельную ветку от зафиксированного `main`
3. Изменить только заявленный WRITE_SCOPE
4. Выполнить локальную валидацию на точном HEAD
5. Создать Draft PR и провести lifecycle review
