# finuslugi-digest-editor

`finuslugi-digest-editor` - личный навык для редакторской проверки продуктовых дайджестов Финуслуг. Он работает в трех режимах: вычитка по редполитике, формирование или аудит тегов и сравнение двух версий по фактам и смыслу.

Навык опирается только на предоставленные материалы и активную редполитику. Он сохраняет точные числа, знаки, единицы, периоды и статусы, отмечает непроверяемые фрагменты и не заполняет пробелы догадками.

## Для кого

Навык подходит редакторам, продуктовым командам, дизайнерам, аналитикам и владельцам дайджеста, которым нужно:

- провести доказательное редакторское ревью;
- проверить структуру, метрики, Tone of Voice и теги;
- сформировать теги для самостоятельных карточек;
- найти потери, добавления и искажения между источником и опубликованной версией;
- проверить несколько форматов одного дайджеста без автоматического запуска полного сравнения.

## Три режима

| Режим | Когда включается | Что проверяет | Основной результат |
|---|---|---|---|
| Вычитка | «вычитать», «провести ревью», «проверить по редполитике», «валидировать» или общее «проверить» | продуктовый результат, бизнес-эффект, пользовательскую пользу, метрики, структуру, теги и Tone of Voice | таблица замечаний по серьезности и короткий итог |
| Теги | «подобрать теги», «сформировать теги», «проверить теги» | количество, доказуемость, конкретность, приоритет смыслов и соответствие активной редполитике | таблица тегов или аудита существующих тегов |
| Сравнение | «сравнить», «сопоставить», «проверить соответствие версий», «найти расхождения» | структуру, факты, метрики, статусы, время, причинность, смысл и теги | вердикт, сводка и таблица расхождений |

Если явно запрошено несколько режимов, навык выполняет их в фиксированном порядке:

```text
сравнение -> вычитка -> теги
```

Несколько приложенных файлов сами по себе не включают режим сравнения.

## Входы и выходы

### Поддерживаемые входы

- текст в сообщении;
- DOCX;
- PDF, включая сканированные документы;
- TXT и Markdown;
- презентации;
- скриншоты и изображения;
- таблицы.

Для каждого файла навык строит доказательную карту: фиксирует формат, роль, расположение разделов и карточек, извлеченные утверждения, метрики, статусы, теги и все нечитаемые или неоднозначные места.

### Результат вычитки

Навык возвращает таблицу:

| # | Раздел | Что не так | Тип проблемы | Серьезность | Почему это проблема | Как поправить |
|---:|---|---|---|---|---|---|

После таблицы он называет самые проблемные разделы, приоритет исправлений, системные ошибки и наиболее сильные разделы. Если нарушений нет, навык прямо сообщает об этом.

### Результат тегов

Для формирования тегов:

| # | Раздел | Краткое содержание карточки | Теги или диагностические маркеры | Логика выбора |
|---:|---|---|---|---|

Для аудита существующих тегов:

| # | Раздел | Исходные теги | Вердикт | Что нарушено | Рекомендуемый набор | Логика замены |
|---:|---|---|---|---|---|---|

По базовому контракту формируются три доказуемых тега. Если подтвержденных смыслов недостаточно, навык использует маркер `— (недостаточно данных)` и не выдумывает недостающий тег.

### Результат сравнения

Сначала навык явно назначает роли файлов, затем дает:

- вердикт: `Соответствует`, `Частично соответствует` или `Не соответствует`;
- количество проверенных смысловых единиц;
- количество критичных, средних и низких расхождений;
- таблицу только по найденным расхождениям;
- список корректно перенесенных элементов, первоочередных исправлений и непроверяемых фрагментов.

## Обязательный workflow

Навык всегда проходит один и тот же цикл:

1. Определяет только явно запрошенные режимы.
2. Читает `references/file-handling.md` и строит доказательную карту материалов.
3. Загружает протоколы выбранных режимов.
4. Извлекает и проверяет текст, числа, знаки, единицы, статусы и расположение карточек.
5. Выполняет режимы в порядке `сравнение -> вычитка -> теги`.
6. Сверяет вывод с исходными материалами и форматом протокола.
7. Выдает полный пользовательский результат, а не оставляет его только в служебном выводе.

## Роли файлов

При сравнении по умолчанию:

- файл 1 - источник;
- файл 2 - проверяемая версия.

Пользователь может назначить роли явно. Если порядок нельзя определить надежно и от него зависит вывод, навык задает один уточняющий вопрос.

При вычитке нескольких представлений одного дайджеста навык выбирает основную проверяемую версию, а остальные использует для контроля целостности. При материальном конфликте он фиксирует риск, но не выбирает «правильный» вариант без первоисточника.

## Активная редполитика

По умолчанию используется встроенная политика из `references/redpolicy.md`.

Если пользователь прикладывает собственную редполитику:

- ее содержательные критерии заменяют встроенные;
- процедура, доказательность, категории серьезности и формат выбранного протокола сохраняются;
- внешние редакторские критерии не примешиваются;
- при конфликте количества тегов без указанного приоритета навык задает один уточняющий вопрос.

## Ограничения

- Навык не подтверждает истинность утверждений во внешнем мире без отдельного независимого источника.
- При работе с одним файлом он проверяет внутреннюю согласованность и доказательность, а не проводит внешний фактчекинг.
- Навык не переписывает весь документ и не изменяет исходные файлы без отдельной просьбы.
- Сокращение или стилистическая переформулировка не считается расхождением, если факты, числа, статус и смысл сохранены.
- Ошибки извлечения текста не выдаются за ошибки исходного PDF или DOCX.
- Нечитаемые, обрезанные и неоднозначные фрагменты отмечаются как непроверяемые, а не восстанавливаются по смыслу.
- Общие заголовки, вводные абзацы и декоративные подписи не считаются самостоятельными карточками.
- Несколько файлов не запускают сравнение автоматически.

## Установка

Репозиторий содержит исходники навыка прямо в корне. В personal skill package входят только файлы исполнения:

```text
finuslugi-digest-editor/
|-- SKILL.md
|-- agents/
|   `-- openai.yaml
`-- references/
    |-- comparison-protocol.md
    |-- file-handling.md
    |-- redpolicy.md
    |-- tagging-protocol.md
    `-- validation-protocol.md
```

`README.md` и `LICENSE` относятся только к публичному репозиторию и не должны попадать в personal skill package.

### Собрать ZIP

Запустите из корня репозитория:

```bash
rm -rf dist
mkdir -p dist/finuslugi-digest-editor/agents
mkdir -p dist/finuslugi-digest-editor/references
cp SKILL.md dist/finuslugi-digest-editor/SKILL.md
cp agents/openai.yaml dist/finuslugi-digest-editor/agents/openai.yaml
cp references/comparison-protocol.md dist/finuslugi-digest-editor/references/comparison-protocol.md
cp references/file-handling.md dist/finuslugi-digest-editor/references/file-handling.md
cp references/redpolicy.md dist/finuslugi-digest-editor/references/redpolicy.md
cp references/tagging-protocol.md dist/finuslugi-digest-editor/references/tagging-protocol.md
cp references/validation-protocol.md dist/finuslugi-digest-editor/references/validation-protocol.md
(
  cd dist
  zip -r finuslugi-digest-editor.zip finuslugi-digest-editor
)
```

Загрузите `dist/finuslugi-digest-editor.zip` через доступный в вашем интерфейсе механизм установки личных навыков.

### Проверить доступность

После установки вызовите навык по имени и передайте небольшой тестовый материал:

```text
Используй @finuslugi-digest-editor. Проверь этот текст по встроенной редполитике: «Запустили улучшения и получили отличный результат».
```

Ожидаемый признак загрузки: ответ использует режим вычитки, не придумывает метрику и предлагает конкретное исправление слабого результата.

## Проверенные примеры

### 1. Вычитка

```text
Используй @finuslugi-digest-editor. Вычитай приложенный PDF по встроенной редполитике. Проверь карточки, метрики, структуру, теги и Tone of Voice. Сортируй замечания по критичности и не переписывай документ целиком.
```

Ожидается таблица замечаний и итог по приоритетам исправления.

### 2. Формирование тегов

```text
Используй @finuslugi-digest-editor. Сформируй по три доказуемых тега для каждой самостоятельной карточки приложенного DOCX. Не назначай теги общим заголовкам и не заполняй недостающие смыслы догадками.
```

Ожидается таблица карточек с логикой каждого тега и диагностическими маркерами при нехватке данных.

### 3. Аудит тегов

```text
Используй @finuslugi-digest-editor. Проверь существующие теги в приложенном дайджесте: количество, доказуемость, конкретность и соответствие тексту карточек. Для несоответствующих наборов предложи замену.
```

Ожидается таблица аудита с вердиктом и рекомендуемым набором.

### 4. Сравнение версий

```text
Используй @finuslugi-digest-editor. Сравни файл 1 как источник и файл 2 как проверяемую версию. Найди потери, добавления, конфликты чисел, изменение статусов, причинности и смысла. Покажи только расхождения и точные значения из обоих файлов.
```

Ожидается явное назначение ролей, общий вердикт и таблица расхождений.

### 5. Все три режима

```text
Используй @finuslugi-digest-editor. Сначала сравни исходный DOCX с финальным PDF, затем вычитай PDF по приложенной редполитике и после этого проверь существующие теги. Раздели результаты по режимам.
```

Ожидаемый порядок: `сравнение -> вычитка -> теги`.

## Структура репозитория

```text
finuslugi-digest-editor/
|-- SKILL.md
|-- README.md
|-- LICENSE
|-- agents/
|   `-- openai.yaml
`-- references/
    |-- comparison-protocol.md
    |-- file-handling.md
    |-- redpolicy.md
    |-- tagging-protocol.md
    `-- validation-protocol.md
```

Назначение файлов:

- `SKILL.md` - точка входа, маршрутизация режимов и общие ограничения;
- `agents/openai.yaml` - отображаемое имя, краткое описание и prompt по умолчанию;
- `references/file-handling.md` - извлечение и доказательная карта материалов;
- `references/validation-protocol.md` - вычитка по активной редполитике;
- `references/tagging-protocol.md` - формирование и аудит тегов;
- `references/comparison-protocol.md` - сравнение структуры, фактов и смыслов;
- `references/redpolicy.md` - встроенная редполитика и Tone of Voice;
- `README.md` и `LICENSE` - repository-only материалы, исключенные из personal skill package.

## Ручная валидация

### 1. Проверить whitespace и diff

```bash
git diff --check
```

### 2. Проверить обязательные файлы

```bash
python3 - <<'PY'
from pathlib import Path

required = [
    "SKILL.md",
    "agents/openai.yaml",
    "references/comparison-protocol.md",
    "references/file-handling.md",
    "references/redpolicy.md",
    "references/tagging-protocol.md",
    "references/validation-protocol.md",
    "README.md",
    "LICENSE",
]

missing = [path for path in required if not Path(path).is_file()]
if missing:
    raise SystemExit("Missing required files: " + ", ".join(missing))

print(f"PASS: {len(required)} required files are present")
PY
```

### 3. Проверить имя навыка и ссылки из SKILL.md

```bash
python3 - <<'PY'
from pathlib import Path

skill = Path("SKILL.md").read_text(encoding="utf-8")
required_references = [
    "references/file-handling.md",
    "references/validation-protocol.md",
    "references/tagging-protocol.md",
    "references/comparison-protocol.md",
    "references/redpolicy.md",
]

if not skill.startswith("---\n"):
    raise SystemExit("SKILL.md: YAML frontmatter is missing")
if "\nname: finuslugi-digest-editor\n" not in skill:
    raise SystemExit("SKILL.md: unexpected skill name")

for reference in required_references:
    if not Path(reference).is_file():
        raise SystemExit(f"Missing reference: {reference}")
    if f"`{reference}`" not in skill:
        raise SystemExit(f"SKILL.md does not reference: {reference}")

print("PASS: skill name and reference links are valid")
PY
```

### 4. Проверить состав personal skill package

```bash
python3 - <<'PY'
from pathlib import Path

package_files = [
    Path("SKILL.md"),
    Path("agents/openai.yaml"),
    Path("references/comparison-protocol.md"),
    Path("references/file-handling.md"),
    Path("references/redpolicy.md"),
    Path("references/tagging-protocol.md"),
    Path("references/validation-protocol.md"),
]
repository_only = {Path("README.md"), Path("LICENSE")}

missing = [str(path) for path in package_files if not path.is_file()]
if missing:
    raise SystemExit("Missing package files: " + ", ".join(missing))
if repository_only.intersection(package_files):
    raise SystemExit("Repository-only files entered the package manifest")

print("PASS: package contains 7 runtime files; README.md and LICENSE are excluded")
PY
```

### 5. Проверить локальные Markdown-ссылки README

```bash
python3 - <<'PY'
import re
from pathlib import Path

readme = Path("README.md").read_text(encoding="utf-8")
links = re.findall(r"\[[^\]]+\]\(([^)]+)\)", readme)
local_links = [link.split("#", 1)[0] for link in links if not link.startswith(("http://", "https://", "#"))]
missing = sorted({link for link in local_links if link and not Path(link).exists()})

if missing:
    raise SystemExit("Broken local README links: " + ", ".join(missing))

print(f"PASS: {len(local_links)} local README links are valid")
PY
```

## Изменения и участие

Изменения поведения навыка должны вноситься отдельно от продуктовой упаковки и сопровождаться проверкой трех режимов. Не меняйте протоколы только ради упрощения README или прохождения примера.

Для предложения изменения:

1. Создайте отдельную Issue с целью, scope, non-goals и acceptance criteria.
2. Работайте в отдельной ветке от зафиксированного commit `main`.
3. Меняйте только файлы в заявленном scope.
4. Выполните ручную валидацию и приложите evidence.
5. Создайте Draft PR и не мержите его до отдельного lifecycle-решения.
