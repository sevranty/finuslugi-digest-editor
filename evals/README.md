# Regression fixtures

Набор проверяет контракт `finuslugi-digest-editor` без изменения `SKILL.md`, редполитики или протоколов. Все материалы синтетические и предназначены только для воспроизводимой ручной регрессии.

## Покрытие

| Case | Режим | Назначение |
|---|---|---|
| `validation-positive` | вычитка | корректная карточка без искусственных замечаний |
| `validation-negative` | вычитка | слабый результат, отсутствие метрики, Tone of Voice и структура |
| `tagging-positive` | теги | три доказуемых смысловых роли |
| `tagging-negative` | теги | нехватка данных и диагностический маркер |
| `comparison-positive` | сравнение | допустимая переформулировка без потери фактов и смысла |
| `comparison-negative` | сравнение | статус, scope, метрика, причинность и удалённое ограничение |
| `combined` | сравнение -> вычитка -> теги | единый порядок режимов и разделение результатов |

Машиночитаемые ожидания находятся в `evals/expected-results.json`. SHA-256 исходных fixtures и expectations зафиксированы в `evals/manifest.json`.

## Структурная проверка

Из корня репозитория выполнить:

```bash
python3 scripts/validate_fixtures.py
```

PASS означает:

- все пути из manifest существуют;
- SHA-256 каждого файла совпадает;
- все case ID уникальны;
- для каждого режима есть позитивный и негативный сценарий;
- combined case использует порядок `comparison -> validation -> tagging`;
- inputs каждого case входят в manifest;
- политика manifest подтверждает синтетические данные и отсутствие изменений runtime-контракта.

## Ручная проверка поведения

Для каждого case передать указанные `inputs` установленному навыку и явно запросить режимы из поля `mode`.

1. Для comparison назначить роли из `roles`.
2. Сверить ответ с assertions из `expected`.
3. Проверить точные числа, знаки, периоды, статусы и ограничения.
4. Не требовать дословного совпадения формулировок, если фактический контракт выполнен.
5. Зафиксировать `PASS`, `FAIL` или `BLOCKED` и конкретное несовпавшее assertion.
6. При изменении fixture обновить его SHA-256 в manifest только после ручного review.

Пример combined-запроса:

```text
Используй @finuslugi-digest-editor. Сравни файл 1 как источник с файлом 2 как проверяемой версией, затем вычитай файл 2 по встроенной редполитике и после этого проверь его существующие теги. Раздели результаты по режимам.
```

## Pass/fail criteria

- `PASS`: все обязательные assertions выполнены, лишние факты не добавлены.
- `FAIL`: пропущено обязательное расхождение, изменено число или статус, выдуман тег/факт либо нарушен формат режима.
- `BLOCKED`: материал не удалось прочитать или runtime недоступен; это не считается продуктовым PASS.

Fixtures, `evals/`, `scripts/`, README и LICENSE являются repository-only материалами и не входят в personal skill package.
